Implementation Plan: Kiểm soát Chi phí & Caching

Tối ưu hóa tài nguyên hệ thống và quản lý ngân sách sử dụng AI LLM.

1. Overview (Tổng quan)

Module Kiểm soát Chi phí & Caching tập trung vào việc tối ưu hóa hiệu năng và tiết kiệm chi phí vận hành bằng cách giảm thiểu số lượng request gửi tới LLM thông qua Semantic Caching. Đồng thời, module này thiết lập các rào cản kỹ thuật để bảo vệ tài nguyên hệ thống (DB/API) khỏi việc bị lạm dụng thông qua Rate Limiting và Tracking.

2. Các bước triển khai chi tiết

Bước 1: Triển khai Semantic Caching với Redis

Mục tiêu: Giảm chi phí LLM bằng cách tái sử dụng các kết quả truy vấn tương tự.

  • Tác vụ 1: Cấu hình Redis service và cache store.
  • Tác vụ 2: Tích hợp logic Similarity Search cho các câu hỏi đã trả lời.
"Hãy thiết kế một lớp Caching cho NL2SQL Service. Sử dụng Redis để lưu trữ cặp 'Question Embedding' và 'SQL Result'. Khi có câu hỏi mới, thực hiện tìm kiếm độ tương đồng. Nếu tìm thấy câu hỏi tương tự (>0.95 similarity), hãy trả về kết quả ngay mà không gọi LLM API. Đầu ra: Cache Layer hoạt động ổn định."

Bước 2: Rate Limiting & Query Timeout

Mục tiêu: Bảo vệ hệ thống khỏi các yêu cầu độc hại hoặc spam.

  • Tác vụ 1: Xây dựng middleware Rate Limit cho Gateway.
  • Tác vụ 2: Cấu hình Timeout ngắt truy vấn database sau 30 giây.
"Thực hiện viết một Middleware cho Gateway API để giới hạn số lượng câu hỏi của mỗi người dùng (Rate Limiting). Ngoài ra, hãy cấu hình Query Service để tự động ngắt (TIMEOUT) các truy vấn SQL chạy quá 30 giây. Mục tiêu: Ngăn chặn spam và treo database."

Bước 3: Token Tracking & Cost Analytics

Mục tiêu: Minh bạch hóa chi phí sử dụng AI theo từng đối tượng.

  • Tác vụ 1: Trích xuất dữ liệu usage từ LLM API response.
  • Tác vụ 2: Lưu token tracking vào bảng logs phục vụ báo cáo Admin.
"Cập nhật logic gọi LLM API để trích xuất số lượng token tiêu thụ. Lưu các thông số này kèm theo user_id vào bảng logs trong database. Xây dựng một view thống kê đơn giản để xem tổng chi phí theo phòng ban. Mục tiêu: Kiểm soát ngân sách sử dụng AI."
